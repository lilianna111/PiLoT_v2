# 深度/角度梯度占比过小分析

## 现象

- 梯度范数：重投影 ≈ 52.37，深度(加权) ≈ 0.00014，角度(加权) ≈ 0.011
- 梯度占比：重投影 ≈ 100%，深度 ≈ 0%，角度 ≈ 0.02%

## 一、深度梯度为什么几乎为 0

### 1. 鲁棒核把深度项压掉了

- 诊断里：**深度残差(原始): mean=144.1499m**
- 验证里：**中心点 z − DSM = 0.33m**（物理上应为 ~0.3m 量级）

说明**参与 loss 的深度残差被算成了 ~144m**，而不是 ~0.33m。

- `cost_depth = (depth_res)^2`，`depth_res = raw_residual / DEPTH_NORM`（DEPTH_NORM=10）
- 若 raw = 144 → depth_res = 14.4 → cost_depth = 207
- 鲁棒核（alpha=0 的 log）：`x = cost / truncate^2 = 207/0.01 = 20700`，权重 `w_loss = 2/(x+2) ≈ 0.0001`
- 深度梯度：`grad_depth ∝ depth_res * w_loss * J_depth` → 被 w_loss 压到约 0.0001 倍，所以 **‖grad_depth‖ ≈ 0.00014**

结论：**深度梯度小是因为“深度残差”被错误地算成 ~144m，鲁棒核把它当离群点，权重大幅缩小。**

### 2. 144m 很可能来自“用错的高度”

- 144 ≈ 181 − 37 ≈ **相机高度(alt) − 地面高度**
- 若某处把 **相机高度** 当成中心点高度 `estimated_z`，而 DSM 取的是地面高度，就会得到 residual ≈ 144m。
- 正确应是：**中心射线落点的 WGS84 高度**（~36m）− DSM（~36m）≈ 0.33m。

建议：

1. 跑一次带新打印的版本，看 `[深度残差来源]` 里 **estimated_z** 和 **h_dsm**：
   - 若 estimated_z ≈ 181 → 说明中心点高度用了相机高度，需要改为中心射线落点高度。
   - 若 estimated_z ≈ 36、h_dsm ≈ 36 → 说明残差本应 ~0.33，若此时 mean(base_residuals_raw) 仍为 144，则可能是别处（如 batch/候选索引）用错。
2. 确认 `center_points` 来自 `_center_points_from_pose_ecef` / `_center_points_from_wgs84_poses` 的**射线落点** (lon, lat, z)，且 `center_points_np[:, 2]` 取的是该 z，而不是相机 alt。

## 二、角度梯度为什么相对小

- 角度残差（弧度）mean ≈ 0.00589 ≈ **0.34°**，本身很小。
- 角度只有 **2 个标量**（roll、pitch），重投影有 **N 个点 × C 维**，项数差几个数量级。
- 因此 **‖grad_angle‖** 自然比 **‖grad_proj‖** 小很多，这不是 bug，而是量纲/项数差异。
- 若希望角度约束更明显，可以：
  - 适当增大可学习权重 `w_angle`，或
  - 在 loss 里对角度项乘一个固定放大系数（需与重投影量纲协调）。

## 三、小结

| 项     | 原因 |
|--------|------|
| 深度≈0% | 深度残差被算成 ~144m → 鲁棒权重大幅下降 → 梯度被压到 ~0.0001 倍；需排查 estimated_z 是否用了相机高度。 |
| 角度≈0.02% | 角度残差小（~0.34°）且只有 2 维，相对数百个重投影项本就小；可考虑增大 w_angle 或角度项权重。 |

修复“深度残差 144m”后，深度项的鲁棒权重会回到正常量级，深度梯度占比会明显上升。
